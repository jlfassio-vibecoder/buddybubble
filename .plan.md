# BuddyBubble Coach — P0/P1 conversational intake (implementation)

## Scope

P0/P1: [`supabase/functions/bubble-agent-dispatch/index.ts`](supabase/functions/bubble-agent-dispatch/index.ts) only. P2/P3: same Edge Function plus [`supabase/migrations/20260531100000_agent_create_card_seed_task_comment.sql`](supabase/migrations/20260531100000_agent_create_card_seed_task_comment.sql) and [`src/types/database.ts`](src/types/database.ts). Slack-style threading (`threadId`, `loadThreadHistory`) is unchanged.

### Deploy order (P2/P3)

1. Apply the migration (`supabase db push` / hosted migrate) so `public.agent_create_card_and_reply` includes `p_seed_task_comment_text` (11-arg signature).
2. Deploy the Edge Function so it passes `p_seed_task_comment_text` only when creating a card.
3. Regenerate or sync `src/types/database.ts` if your workflow differs from the committed Args update.

## What changed

### P0 — `fetchUserContext` and last workout

- **Completed status:** Last-workout queries use `status` in `('done', 'completed')` (aligned with board semantics).
- **Columns:** Last-workout selects include `metadata` and `description` for summaries.
- **Ordering:** `scheduled_on` descending (nulls last), then `created_at` descending.
- **User vs bubble:** Query `assigned_to = userId` first; if no titled row, fall back to bubble-wide last completed. Logs `last_workout_scope=assigned|bubble`.
- **Helpers:** `summarizeWorkoutTaskMetadata`, `bestEffortCompletedAtLabel` (no `tasks.completed_at` column — uses optional metadata keys `completed_at` / `session_completed_at` / `finished_at`, else `scheduled_on`, else `created_at`), `truncateOneLine` for description fallback.
- **Prompt:** Appends `--- LAST WORKOUT CONTEXT ---` (Title, `completed_at (best effort)`, Summary) after `--- CURRENT USER CONTEXT ---` when a last completed workout exists.

### P1 — Gemini schema, prompt, parsing

- **`responseSchema`:** Added `intake_phase`, `session_readiness_score`, `missing_intake_categories`, `user_requested_immediate_card`. Existing `required` keys unchanged so the model still returns `task_title` / `task_description` every turn.
- **`CoachGeminiJsonResponse`:** Extended return type from `parseGeminiJsonText` / `geminiGenerateJson`.
- **Parsing defaults:** Invalid or missing intake fields → `intake_phase: 'other'`, `session_readiness_score: 0`, `missing_intake_categories: []`, `user_requested_immediate_card: false` unless JSON boolean `true`.
- **`baseCoachPrompt`:** Consultative-first: session readiness vs profile; do not create a card until `missing_intake_categories` is empty (or explicit fast path); use `LAST WORKOUT CONTEXT` for grounded follow-ups; JSON key list updated.

### P1 — Layer B (server policy)

Superseded for turn counting by **P2** below; first-message block remains part of Layer B.

### P2 — Layer B turn gating (Edge)

- **`session_request`:** New Gemini boolean; default `false` unless JSON `true`. Model marks when the user wants a concrete workout/session soon.
- **Rules** (skipped when `user_requested_immediate_card` is true):
  - **`first_message_card_blocked`:** `priorUserMessageCount === 0` → force no card, clear `task_title` / `task_description` / seed notes.
  - **`session_request_turn_gate`:** `session_request && priorUserMessageCount < 2` → same force (blocks the **2nd** user turn for session intents; **3rd** user turn is `priorUserMessageCount >= 2`).
- **`priorUserMessageCount`:** Count of prior rows in `historyAsc` whose `user_id` is not an agent auth id.
- **Logging:** JSON line includes `layerB`, `priorUserMessageCount`, `session_request`, `model_create_card`, and intake fields.

| Prior user msgs | `session_request` | `user_requested_immediate_card` | Card allowed (model `create_card` true) |
| --------------- | ----------------- | ------------------------------- | --------------------------------------- |
| 0               | any               | false                           | No (`first_message_card_blocked`)       |
| 1               | true              | false                           | No (`session_request_turn_gate`)        |
| 1               | false             | false                           | Yes (if model allows)                   |
| ≥2              | true              | false                           | Yes (if model allows)                   |
| any             | any               | true                            | Yes (if model allows; Layer B skipped)  |

### P3 — Coach task notes (Edge + DB)

- **Gemini:** `coach_task_notes` nullable string; capped in the Edge Function at **8000** characters before RPC.
- **Prompt:** When `create_card` is true, fill `coach_task_notes` with readiness summary, rationale, and scaling options; `task_description` stays the executable card body.
- **RPC:** New optional `p_seed_task_comment_text text default null`. After inserting the new `tasks` row, if trimmed text is non-empty, insert `public.messages` with `target_task_id = v_task_id`, `user_id = p_agent_auth_user_id`, `parent_id` null, `attached_task_id` null (task-scoped comment; triggers maintain `comment_count`).
- **Edge RPC payload:** Pass `p_seed_task_comment_text` when creating a card; clear when Layer B strips the card.

## Verification

1. **Edge logs (Supabase):** Send first `@Coach` message in a new thread without “put it on a card” language. Expect a JSON log with `layerB: "first_message_card_blocked"` and `model_create_card` possibly `true`; RPC should not create a card (`p_create_card` false).
2. **Second message:** Reply in the same thread with explicit “just put it on a card” / “generate it now”; set `user_requested_immediate_card` via model; Layer B must not run (`priorUserMessageCount > 0`). Card creation may proceed when the model sets `create_card: true` with a valid title.
3. **Last workout block:** Mark a workout or workout_log done with `metadata.exercises`; confirm coach questions can reference exercise names (avoid logging full PHI in production; context length debug log remains).
4. **P2:** Third user message in a `session_request` thread may create a card; second message logs `session_request_turn_gate` if the model still requests a card. Immediate-card language bypasses gating.
5. **P3:** After card creation, query `messages` for `target_task_id = created_task_id` and agent `user_id`; confirm task comments UI shows the seed. Confirm RPC signature has 11 arguments in Postgres.

- Local `deno check` / tests if Deno is installed; not required for deployment.

# Vertex — Extract & Enrich pipeline (Authoritative Kanban briefs)

## Goal

When `persona.kanbanBriefAuthoritative === true` and the workout is **not** a metabolic special mode (HIIT / density AMRAP / balanced Tabata), `POST /api/ai/generate-workout-chain` uses a **two-step Vertex path** instead of Architect → Biomechanist → Coach → Mathematician. This preserves the Coach/Kanban brief and fills three detail tiers for `tasks.metadata.exercises`.

## Branch condition

- **Use Extract & Enrich:** `kanbanBriefAuthoritative` and at least one of `title` / `description` trimmed, and **none** of `hiitMode`, `amrapDensityMode`, `tabataBalancedMode`.
- **Use legacy 4-step:** all other requests (including authoritative + metabolic modes — phase 2 if needed).

## Architecture

1. **Step A — Extract** ([`src/lib/workout-factory/prompt-chain/extract-workout-from-brief.ts`](src/lib/workout-factory/prompt-chain/extract-workout-from-brief.ts)): `title` + `description` → strict JSON (`exercises[]` with `order`, `section`, `exercise_name`, `sets`, `reps`, `equipment`, optional `rest_seconds`, `rpe`, timer fields). Rule: do not invent or rename movements; do not split into multiple days.
2. **Step B — Enrich** ([`src/lib/workout-factory/prompt-chain/enrich-workout-biomechanics.ts`](src/lib/workout-factory/prompt-chain/enrich-workout-biomechanics.ts)): Step A JSON + optional medical context → `detailed_instructions`, `biomechanical_cues`, `injury_prevention_tips` per `order` (immutable prescription fields from Step A).
3. **Assembly** ([`src/lib/workout-factory/map-kanban-extract-to-workout.ts`](src/lib/workout-factory/map-kanban-extract-to-workout.ts), [`src/lib/workout-factory/generate-workout-kanban-extract-runner.ts`](src/lib/workout-factory/generate-workout-kanban-extract-runner.ts)): merge → `WorkoutSetTemplate` + `taskExercises`; [`normalizeWorkoutSet`](src/lib/workout-factory/program-schedule-utils.ts) for canonical blocks.
4. **Router** ([`src/lib/workout-factory/generate-workout-chain-runner.ts`](src/lib/workout-factory/generate-workout-chain-runner.ts)): `runGenerateWorkoutChain` delegates to `runExtractAndEnrichChain` when the branch matches.

## Storage (no new SQL table)

Per-exercise data lives in **`tasks.metadata.exercises`** as [`WorkoutExercise`](src/lib/item-metadata.ts). Extended optional fields: `equipment`, `injury_prevention_tips` (plus existing `instructions`, `form_cues` for detailed / highly detailed tiers).

## `chain_metadata`

Discriminated union on `pipeline`: `legacy_four_step` | `kanban_extract_enrich`. Legacy responses set `pipeline: 'legacy_four_step'`. Kanban path stores extract + enrich payloads under typed keys (see [`src/lib/workout-factory/types/ai-workout.ts`](src/lib/workout-factory/types/ai-workout.ts)).

## API route

[`src/app/api/ai/generate-workout-chain/route.ts`](src/app/api/ai/generate-workout-chain/route.ts): if `result.data.taskExercises` is present, return it; else derive from `workoutInSetToTaskExercises(firstWorkout)` (legacy).

## Verification

1. Kanban workout card with rich markdown description → Generate → `chain_metadata.pipeline === 'kanban_extract_enrich'`; exercise **names** match the brief; `taskExercises[].instructions` / `form_cues` / `injury_prevention_tips` populated where the model returns them.
2. Non-authoritative or HIIT profile request → still `legacy_four_step` and four Vertex steps.
3. Save task → `metadata.exercises` round-trips in Task Modal / workout viewer.

## Exercise dictionary cache-aside (Kanban)

**Flow:** After Step A validates, the Kanban runner calls `createServiceRoleClient` and `exercise_dictionary_lookup_by_names(p_names text[])` (batch RPC, `service_role` execute only). Rows are matched with `lower(trim(name))` against extract names; only `status in ('published','pending')`. **Tie-break:** if multiple rows could match one name, Postgres returns one row per normalized name — prefer `published`, then newest `updated_at` (`DISTINCT ON` in [`supabase/migrations/20260618120000_exercise_dictionary_lookup_by_names.sql`](supabase/migrations/20260618120000_exercise_dictionary_lookup_by_names.sql)).

**Split:** [`splitExtractByDictionaryMatches`](src/lib/workout-factory/exercise-dictionary-bridge.ts) builds `foundByOrder` vs `missing[]`. Step B prompt uses a **partial** extract JSON when `missing.length` is between 1 and N−1 ([`buildEnrichWorkoutBiomechanicsPrompt`](src/lib/workout-factory/prompt-chain/enrich-workout-biomechanics.ts) `extractIsPartialSubset`); [`validateEnrichWorkoutBiomechanicsOutput`](src/lib/workout-factory/prompt-chain/enrich-workout-biomechanics.ts) checks length against that subset only.

**Skip Vertex:** If `missing.length === 0`, Step B is skipped; [`mergeKanbanEnrichFromDictionaryAndVertex`](src/lib/workout-factory/exercise-dictionary-bridge.ts) builds full `KanbanEnrichBiomechanicsOutput` from dictionary rows only, then [`mergeKanbanExtractEnrichToTaskExercises`](src/lib/workout-factory/map-kanban-extract-to-workout.ts) unchanged.

**Degrade:** If `SUPABASE_SERVICE_ROLE_KEY` / URL setup throws, or the lookup RPC throws, the runner logs `exercise_dictionary_cache_skipped` and runs Step B on the **full** extract (full Vertex enrich).

**Write-back:** After a successful Step B (or full degrade path), for each order that was enriched via Vertex (`missing` set), the runner inserts `exercise_dictionary` with `status: 'pending'`, `slug` from [`slugifyExerciseName`](src/lib/workout-factory/exercise-dictionary-bridge.ts) + unique suffix, `instructions` / `biomechanics` from enrich fields. Insert errors log `exercise_dictionary_insert_failed` and do **not** fail the HTTP response.

**Verification:** Seed one `exercise_dictionary` row whose normalized `name` matches an extract line → that slot is filled from DB and Step B is skipped when all slots hit the cache; generate a never-seen movement name → a new `pending` row appears after generation (when service role is configured).

# V2: Coach updates an existing card (Edge + Postgres)

## Deploy order (critical)

1. Apply the migration [`supabase/migrations/20260617120000_agent_update_task_and_reply.sql`](supabase/migrations/20260617120000_agent_update_task_and_reply.sql) so `public.agent_update_task_and_reply` exists (`service_role` execute only).
2. Deploy the [`bubble-agent-dispatch`](supabase/functions/bubble-agent-dispatch/index.ts) Edge Function.

Deploying the function before the migration causes `rpc_failed` when the update path runs.

## Postgres RPC

- **`agent_update_task_and_reply`:** Updates `tasks.title` / `tasks.description` with `coalesce(p_new_*, …)`; inserts the agent bubble reply with `parent_id = p_thread_id`, `attached_task_id = p_target_task_id`; same advisory lock + idempotency + invoker/agent/bubble checks as [`agent_create_card_and_reply`](supabase/migrations/20260531100000_agent_create_card_seed_task_comment.sql).
- **Idempotency:** Reuses `agent_message_runs`; on success sets `created_task_id = p_target_task_id` for this path (related task id).
- **Validation:** At least one of `p_new_title` / `p_new_description` must be non-empty after trim.

## Edge Function

- **Gemini schema:** `update_existing_task` (boolean), `updated_task_title`, `updated_task_description` (nullable strings); required alongside existing keys.
- **`known_target_task_id` (server only):** Resolved from the trigger `messages` row (`target_task_id`, `attached_task_id`) then recent thread history (same columns), validated against `tasks` in the bubble. Never sent to or read from the LLM.
- **System prompt:** When a task is resolved, inject `CURRENT TASK CONTEXT` (title + description) instructing the model to set `update_existing_task` and full revised fields when the user asks for edits.
- **Routing:** If `update_existing_task && knownTargetTaskId &&` at least one updated field → `agent_update_task_and_reply`. Else if `create_card` → `agent_create_card_and_reply` (create). Else → `agent_create_card_and_reply` with `p_create_card: false`. Prefer update over create when both would apply (`create_card` forced false). Layer B still clears new-card fields but does not strip update intent; conflict block after Layer B re-applies “prefer update”.
- **Fallback:** If the model sets `update_existing_task` but provides no title/description body, log `update_existing_task_ignored_no_body` and use the reply-only create RPC.

## Types

[`src/types/database.ts`](src/types/database.ts) includes `agent_update_task_and_reply` Args/Returns.

## Verification

1. Open a thread with a message scoped to an existing workout task (`target_task_id` / `attached_task_id`) or history containing such a message.
2. Ask Coach to change part of the workout; confirm `tasks` row updates and an agent `messages` row appears in the thread.
3. Retry webhook delivery: expect `deduped: true` from RPC, no duplicate edits.
